## 一、实验目的

本次课程设计要求我独立完成一个具备实际功能的软件项目，并贯穿软件工程的核心环节：需求分析、概要设计、详细设计、编码实现、测试与文档。我希望借这次机会做一件自己真正想用的东西，而不是为了交作业写一个用一次就丢的 demo。

最终我把目标设定为：

> 做一个**带 Agent 能力的个人知识库助手**——能聊天、能记住自己看过的资料、能在对话里回忆、能调用工具完成事情。

围绕这个目标，本次实验的具体目的有四条：

1. 理解并应用一个真实开源项目的二次开发流程，包括架构裁剪与功能扩展
2. 把"个人知识库"从概念落地为一个可用的混合检索系统（向量 + 稀疏 + 融合）
3. 实现 Agent 工具循环，让 LLM 不止"回答"还能"做事"
4. 用工程化方式管理代码、文档、构建产物，符合软件工程课程对项目交付的要求

## 二、需求分析

### 2.1 用户场景

我自己每天会在不同的对话窗口里和 LLM 聊天，但每次开新窗口它都不认识我。我需要一个能记住我看过哪些资料、聊过哪些话题的助手。

由此推导出三个核心场景：

- **场景 A：上传资料后能查得到** —— 我把课程 PPT、论文、技术文档丢进去，下次提问时它要能从里面引用
- **场景 B：聊过的事不要忘** —— 我说过我在做什么项目、用什么技术栈，下次问相关问题时它要带着上下文回答
- **场景 C：让它去做事，不只是回答** —— 调用工具、查文件、执行简单操作

### 3.2 关键设计决策

#### 决策一：在 AstrBot 上做减法而非从零写

我对比了"从零写"和"基于 AstrBot 二开"两条路。从零写的好处是干净，坏处是 LLM Provider 适配、Pipeline 调度、SSE 这些工程细节我得自己重做一遍，3 周做不完。

最终选择 fork AstrBot，砍掉 13 个我用不到的平台（QQ/微信/Telegram/Discord 等），只留 WebChat。又砍掉了子 Agent 编排栈、多用户配置、ApiKey CRUD 等用不到的模块。**最终保留下来的代码反而更聚焦。**

#### 决策二：双知识库共享同一个检索引擎

我一开始打算做两套独立的知识库系统：一套放用户上传的资料，一套放对话沉淀。后来意识到这两件事本质上都是"语义检索 + 召回"，没必要分两套。

最后的设计是：

- **同一个 KB Helper** —— FAISS + BM25 + RRF 融合
- **两个 collection** —— 外接库 / 对话库各自隔离
- **检索时区分注入位置** —— 对话库进入「关于用户的相关信息」段，外接库进入「可能相关的资料」段

这样 LLM 在 system_prompt 里就天然能区分"这是我已知的用户事实"和"这是用户给我的参考资料"。

#### 决策三：Agent 工具循环用 ReAct 模式

LLM Provider 已经原生支持 function calling，我没必要自己设计 prompt 协议。直接用 ReAct：调用 → 执行 → 结果回灌 → 继续，最多 N 轮。工具集就是一个注册表，KB 检索 / 网络搜索 / Computer Use / MCP 都注册成 function。

## 四、详细设计与实现

### 4.1 知识库模块

```
miku/core/knowledge_base/
├── kb_helper.py            # 统一入口
├── kb_manager.py           # collection 生命周期
├── retrieval/
│   ├── dense.py            # FAISS 向量检索
│   ├── sparse.py           # BM25 + 中文分词
│   └── fusion.py           # RRF 融合
├── personal_kb.py          # 对话知识抽取
└── document_loader.py      # PDF/DOCX/MD/EPUB 解析
```

**RRF 融合**采用经典公式 `score = sum(1 / (k + rank_i))`，参数 k=60。这种融合不依赖各路打分的绝对值，对向量分数和 BM25 分数的混合特别合适。

**对话知识抽取**用一个固定 system_prompt 让 LLM 把对话切成八类条目（兴趣 / 项目 / 技术栈 / 个性 / 学习 / 偏好 / 概念 / 关系），每条带 confidence。低于 0.3 的丢弃。

### 4.2 Agent 工具循环

```python
async def run(self, ...):
    while step < max_steps:
        response = await llm.chat(messages, tools=tools)
        if response.tool_calls:
            for call in response.tool_calls:
                result = await tool_executor.invoke(call)
                messages.append(tool_result_to_message(call, result))
        else:
            return response.content
        step += 1
```

工具注册靠装饰器：

```python
@tool_executor.register
async def kb_search(query: str, top_k: int = 5):
    ...
```

### 4.3 数据流

用户消息 → Pipeline → 知识库自动检索 → 注入 system_prompt → LLM → 工具调用循环 → 流式返回前端 → 持久化到 SQLite

### 4.4 一键部署

`setup.bat` 流程：检测 Python 3.12 → 创建 venv → `pip install -r requirements.txt` → `npm install && npm run build` → 拷贝 dashboard/dist 到 data/dist。

`_run_miku.bat` 流程：检测 venv → 检测端口 → 检测 aiohttp → `python main.py`。

### 5.3 缺陷与修复

最具代表性的一个 bug：**对话知识抽取报"没有足够的对话数据"**。

排查过程：

1. 看到错误以为是真的对话太少，但加了几十条还是报错
2. 读代码发现 `profile.py` 里 `if not conversations_text or not conversations_text.strip()` 才报错，意味着收集到的字符串是空的
3. 继续往上看 `_gather_session_history` 用的是 `conversation_mgr.get_conversations()`，访问 `conv.session_id`
4. 翻 `Conversation` 数据类发现根本没有 `session_id` 字段，只有 `cid` 和 `user_id`，访问会抛 AttributeError 被外层 try/except 吞掉
5. 进一步发现前端会话下拉数据来自 `/api/miku/sessions`，对应的是 `ConversationStore`（独立 SQLite）——和 `conversation_mgr` 是两个不同的库

根因：**两个数据源不同**，前端选的 session_id 在后端找不到对应的 conversation，于是永远返回空字符串。

修复（commit `7218d2e`）：把 `_gather_session_history` 和 `_gather_recent_history` 都改用 `ConversationStore`，和前端会话下拉对齐。

## 七、收获与反思

### 7.1 学到了什么

**减法比加法更难。** AstrBot 是一个非常完整的多平台框架，但它什么都做意味着对个人知识库这件事的支持反而很浅。我花了很多时间在"砍"——砍平台、砍子 Agent 栈、砍主题、砍多用户体系。每砍一刀，剩下的代码反而更清晰。

**KB 和 Agent 不应该分两套系统。** 把对话沉淀（画像）和外接资料（文档）用同一个检索引擎承载，让 Agent 工具直接调用同一个 KB Helper，整个系统的边界一下子就清楚了——一个能记住、能查、能动手的助手，本质上就是这三件事的统一表达。

**工程化的文档和提交记录在评审时是真的有用。** 我后期补救式地把改动拆成语义 commit，README 重写，做了项目评审文档和实验报告，这些工作量在写的时候觉得繁琐，但回头看，没有它们我自己都说不清这 23 天到底做了什么。

### 7.2 做得不够好

| 方面 | 问题 |
|------|------|
| 测试覆盖 | 几乎全靠手测和集成测试，单元测试缺位 |
| 提交策略 | 早期单 commit，到后期才拆分，前半段历史不够干净 |
| 跨平台 | 只在 Windows 验证，Linux/macOS 部署脚本没写 |
| Embedding 默认体验 | 知识库强依赖外部 Embedding Provider，离线场景需要手动配 Ollama |
| 自动化 | 没有 CI/CD，每次手动 build |

### 7.3 如果重做

我会从第一天就做三件事：

1. 用语义化 commit + feature branch
2. 至少给 KB 检索和 Agent 工具循环写集成测试
3. 内置一个零依赖的本地 embedding 方案（如 sentence-transformers 小模型），降低评审者门槛

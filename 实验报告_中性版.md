## 一、实验任务与目标

本课程设计要求学生独立完成一个具备完整生命周期的软件工程项目，覆盖需求分析、系统设计、编码实现、集成测试与文档撰写五个环节。

本实验选定的题目为"YUNGE — AI 个人知识库助手"，定位为面向单用户的本地化 AI 助手系统，具备以下三类核心能力：

1. 多模型 LLM 的对话交互
2. 个人知识库的构建、检索与召回
3. 工具调用驱动的 Agent 行为

技术上采用基于开源框架 [AstrBot](https://github.com/AstrBotDevs/AstrBot)（AGPLv3）的二次开发路线，重点在于架构裁剪与功能扩展。

## 三、需求分析

### 3.1 业务背景

通用 LLM 服务在跨会话场景下缺少对用户的持续记忆能力，且无法直接利用用户本地资料进行回答。本系统旨在弥补这一不足，将"长期记忆"与"资料检索"统一纳入对话流程。

### 3.2 功能性需求

依据 MoSCoW 原则划分优先级：

- **Must Have**：WebChat 对话、Provider 多模型接入、文档上传与混合检索、对话历史持久化、对话知识自动抽取、Agent 工具调用循环
- **Should Have**：MCP 协议接入、定时任务、用户画像分析（MBTI / 时间线）、一键部署脚本、数据备份恢复
- **Could Have**：Computer Use 沙箱
- **Won't Have**（本次不交付）：语音合成、移动端、插件市场

### 3.3 非功能性需求

- 单机部署，零外部数据库依赖
- 后端全异步，避免阻塞 SSE 流
- 前后端解耦，REST + 流式接口分离
- 前端构建产物可独立分发，部署阶段不强依赖 Node 工具链

## 四、系统设计

### 4.1 总体架构

系统采用三层架构：

```
┌────────────────────────────────────────┐
│           展示层（Vue 前端）            │
│   WebChat / KB 管理 / 画像 / 配置       │
└────────────────────┬───────────────────┘
                     │ REST + SSE
┌────────────────────┴───────────────────┐
│           业务层（Quart 后端）          │
│   Pipeline · Agent · KB · Provider      │
└────────────────────┬───────────────────┘
                     │
┌────────────────────┴───────────────────┐
│              持久层（SQLite）           │
│      会话 · KB 元数据 · 用户配置         │
└────────────────────────────────────────┘
```

### 4.2 模块划分

| 模块 | 职责 |
|------|------|
| Provider Manager | 统一管理 Chat / Embedding / Rerank / STT 等模型适配器 |
| Pipeline | 消息处理流水线，串联 KB 检索 → Provider → 工具循环 → 输出 |
| Agent Runtime | 实现 ReAct 模式的工具调用循环；接入 MCP Client |
| Knowledge Base | 文档解析、向量化、混合检索；外接库与对话库共享引擎 |
| Conversation Store | WebChat 历史的持久化存储 |
| Profile Service | 基于历史对话的 MBTI 分析与时间线生成 |
| Dashboard Routes | RESTful API 入口 |

### 4.3 关键设计

#### 4.3.1 双知识库共享检索引擎

外接知识库（用户资料）与对话知识库（LLM 抽取的画像条目）虽语义不同，但底层均为"语义检索 + 召回"问题，因此共享同一 KBHelper 实现。

```
        [KBHelper：FAISS + BM25 + RRF]
              │            │
       ┌──────┘            └──────┐
       ▼                          ▼
  外接库 collection          对话库 collection
  （PDF/MD/DOCX...）          （兴趣/项目/技术栈...）
```

每轮对话生成 system_prompt 时，按段落语义分别注入：

- 「关于用户的相关信息」段 ← 对话库（top_k=3，自动）
- 「可能相关的资料」段 ← 外接库（用户启用）

LLM 据此区分用户事实与参考资料。

#### 4.3.2 混合检索策略

向量检索的语义能力强但对关键字精确匹配弱，BM25 反之。本系统采用 RRF（Reciprocal Rank Fusion）融合两路结果：

```
score(d) = Σ 1 / (k + rank_i(d))    k=60
```

可选地接入 reranker 对融合后的 top-N 做重排序。

#### 4.3.3 Agent 工具调用循环

采用基于 LLM function calling 原生能力的 ReAct 循环：

```
loop step in 1..max_steps:
    msg = llm.chat(messages, tools=tools)
    if msg.tool_calls:
        for call in msg.tool_calls:
            result = tool_executor.invoke(call)
            messages.append(format_tool_result(call, result))
    else:
        return msg.content
```

工具集合通过装饰器注册，知识库检索、网络搜索、Computer Use、MCP 工具均以统一接口暴露给 LLM。

### 4.4 数据库设计

主要数据表：

| 表 | 用途 |
|----|------|
| sessions | 会话元信息 |
| conversation_messages | WebChat 消息明细 |
| kb_collections | 知识库 collection |
| kb_documents | 文档元信息 |
| kb_chunks | 切片元数据（向量 ID 在 FAISS） |
| personal_kb_entries | 对话抽取的画像条目 |
| user_profile | MBTI / 时间线缓存 |

## 五、实现要点

### 5.1 平台层裁剪

上游 AstrBot 适配 14 类即时通讯平台。本项目仅需 WebChat，因此移除其余 13 类适配器及消息抽象，将多平台框架降维为 Web 单端应用。配套移除子 Agent 编排栈、多用户体系、ApiKey CRUD 等不必要模块。

### 5.2 知识库实现

- 文档解析：PDF（pypdf）、DOCX（python-docx）、Markdown / TXT、EPUB（ebooklib）、图片（OCR 可选）
- 切分：按 token 与段落混合切分，保留段落边界
- 向量化：通过 Provider Manager 调用任意 Embedding 模型
- 持久化：FAISS index 落盘 + SQLite 存元数据

### 5.3 对话知识抽取

由 LLM 按固定 prompt 将历史会话切分为八类条目：兴趣、项目、技术栈、个性、学习、偏好、概念、关系。每条带置信度（confidence），低于阈值丢弃。抽取结果写入 personal_kb，与外接库共享检索通路。

### 5.4 一键部署

提供两个批处理脚本：

- `setup.bat`：自动检测 Python 3.12 → 创建虚拟环境 → 安装 Python / Node 依赖 → 构建前端 → 部署产物
- `_run_miku.bat`：启动前自检（虚拟环境、端口、关键依赖），随后启动后端服务

为降低评审环境依赖，前端构建产物 `dashboard/dist` 已纳入仓库版本管理。

### 6.3 典型问题与修复

**问题描述**：对话知识抽取功能在选定特定会话后报"没有足够的对话数据"，无法完成抽取。

**根因分析**：`profile.py` 中 `_gather_session_history` 通过 `conversation_mgr.get_conversations()` 获取数据，并按 `conv.session_id` 过滤。然而 `Conversation` 实体上不存在 `session_id` 字段（实际字段为 `cid` / `user_id`），属性访问异常被外层 try/except 吞噬，导致返回空字符串，触发"对话数据不足"分支。前端会话下拉则来自另一份数据源 `ConversationStore`（按 `session_id` 索引），与后端取数据源不一致。

**修复方案**：将 `_gather_session_history` 与 `_gather_recent_history` 重构为统一调用 `ConversationStore`，与前端 `/api/miku/sessions` 数据源对齐。

**修复 commit**：`7218d2e fix(kb): align knowledge extraction & deletion with ConversationStore`

## 八、实验总结

本次课程设计完成了一个面向单用户的 AI 个人知识库助手。系统在功能完整度、模块解耦度与部署便利性三方面均达到课程设计预期目标。

**主要收获**：

1. 在真实开源项目上做二次开发，体验了"做减法"对软件可维护性的正面影响。原生 AstrBot 框架庞大而通用，经过有目的的裁剪后，目标场景下的代码反而更聚焦。
2. 将"个人知识库"与"对话沉淀"统一到同一检索引擎，是设计上的关键决策。两类语义共享底层、分离上层注入位置，使系统边界清晰、扩展简单。
3. Agent 工具循环借助 LLM 原生 function calling，避免了自定义 prompt 协议带来的脆弱性，工具集合可以以统一接口注册扩展。

**改进方向**：

- 引入自动化测试框架（pytest + playwright）覆盖关键路径
- 提供跨平台部署脚本（Linux / macOS）
- 知识库支持版本管理与增量索引
- 内置离线 Embedding 方案以降低依赖门槛

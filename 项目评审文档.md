## 一、项目概述

### 1.1 项目简介

YUNGE 是一款面向个人的 AI 助手，定位"对话即知识沉淀"。后端基于开源框架 [AstrBot](https://github.com/AstrBotDevs/AstrBot) v4.23.6 进行深度二次开发，裁剪到单用户、单平台（WebChat）场景，专注三件事：

1. **聊得清楚** —— 多模型流式对话，支持 Persona、上下文压缩
2. **记得住** —— 外接资料与对话历史共享同一检索引擎，未来对话自动召回
3. **会动手** —— 通过工具调用循环、MCP 协议、定时任务和可选的 Computer Use 沙箱，让助手能完成实际操作

项目目标只有一句话：**做一个带 Agent 能力的个人知识库助手**。

### 2.3 关键技术决策

#### 2.3.1 平台层降维

```
原生 AstrBot（14 平台）          YUNGE（1 平台）
┌──────────────────┐            ┌──────────────┐
│ QQ / 微信 / TG    │ ❌ 删除    │              │
│ Discord / 钉钉    │ ❌ 删除    │   WebChat    │
│ 飞书 / Slack 等   │ ❌ 删除    │   （唯一）   │
│ WebChat           │ ✅ 保留    │              │
└──────────────────┘            └──────────────┘
```

效果：将多平台消息抽象层压扁为 Web 单端应用，运行时复杂度大幅降低，而 Provider / Agent / KB 等核心能力完整保留。

#### 2.3.2 双知识库共享检索引擎

外接知识库（用户上传的资料）与对话知识库（LLM 抽取的画像条目）底层共用 FAISS + BM25 + RRF 融合，但语义上完全分离。

```
              检索引擎（KB Helper）
        ┌──────────────────────────────┐
        │  FAISS dense + BM25 sparse   │
        │  + RRF 融合 + 可选 rerank    │
        └────┬────────────────────┬────┘
             │                    │
   ┌─────────┴────────┐  ┌────────┴─────────┐
   │  外接库（资料）   │  │ 对话库（画像）   │
   │  PDF/DOCX/MD…    │  │ LLM 自动抽取八类 │
   └──────────────────┘  └──────────────────┘

每轮对话注入 system_prompt：
├─ 「关于用户的相关信息」段：来自对话库（top_k=3，自动）
└─ 「可能相关的资料」段：来自外接库（用户启用）
```

LLM 据此区分"用户事实"与"参考资料"，避免互相稀释。

#### 2.3.3 Agent 工具循环

```
用户消息
   │
   ▼
[ToolLoopAgentRunner]
   │  ←── tool_executor 注册的工具集（KB 检索 / 联网 / Computer Use / MCP）
   │
   ├─→ LLM 决策（function calling）
   │      │
   │      ├─ 直接回复 ─→ 输出
   │      │
   │      └─ 调用工具 ─→ 执行 ─→ 结果回灌 ─→ 继续循环
   │
   最多 N 轮（可配置）
```

#### 2.3.4 新增 / 重写的功能模块

| 模块 | 路径 | 说明 |
|------|------|------|
| Agent 工具循环 | `miku/core/agent/tool_loop_agent_runner.py` | 多轮 ReAct，支持工具发现与回灌 |
| MCP 客户端 | `miku/core/agent/mcp_client.py` | 接入 Model Context Protocol |
| 知识库引擎扩展 | `miku/core/knowledge_base/` | RRF 融合 + 中文分词 + 个人 KB |
| 对话知识抽取 | `miku/core/knowledge_base/personal_kb.py` | LLM 抽取八类画像条目 |
| 会话存储 | `miku/core/conversation_store.py` | WebChat 历史持久化 |
| 画像 API | `miku/dashboard/routes/profile.py` + `miku_api.py` | MBTI + 时间线 + 知识抽取入口 |
| 一键部署 | `setup.bat` + `_run_miku.bat` | venv 创建、依赖安装、前端构建、启动 |

### 2.4 技术难点与解决方案

| 难点 | 解决方案 |
|------|---------|
| 全异步框架下的工具循环编排 | 复用 AstrBot Pipeline，自实现 ToolLoopAgentRunner，事件驱动调度 |
| 双 KB 共享引擎但语义隔离 | KBHelper 抽象层 + 调用方分别注入不同 system_prompt 段 |
| 中文 RAG 检索效果 | rank_bm25 + 中文分词 + 停用词表 + RRF 融合，弥补纯向量的语义偏差 |
| 知识抽取数据源对齐 | 修复 profile.py 错误使用 conversation_mgr 的 bug，统一改用 ConversationStore |
| 前端构建产物分发 | dashboard/dist 纳入 git，反向加速课设评审环境部署 |
| 嵌入服务依赖 | 文档中给出云端（OpenAI 兼容 / Gemini）+ 本地（Ollama nomic-embed-text）双方案 |

## 五、创新点与亮点

### 5.1 创新点

1. **双 KB 共享引擎，语义分离** — 同一 FAISS + BM25 索引同时承载用户资料与对话画像，但通过 system_prompt 注入位置区分，避免事实污染。
2. **对话沉淀的画像系统** — 不只是聊天记录持久化，而是 LLM 主动抽取八类条目，与 RAG 共享检索通路，每轮对话自动召回。
3. **Agent + KB + 主动调度三位一体** — 工具循环可调用 KB 检索；定时任务能主动触发 Agent；MCP 让外部工具即插即用。
4. **课设级一键部署** — `setup.bat` + `_run_miku.bat` 两步从干净 Windows 到可运行，构建产物已入库，评审者无需 Node 工具链。

### 5.2 工程亮点

| 亮点 | 说明 |
|------|------|
| 全异步 | Python asyncio + Quart + aiohttp，前后端 SSE 全程非阻塞 |
| 流式 SSE | LLM 输出逐字呈现，体验流畅 |
| venv 隔离 | 不污染系统 Python，迁移即复制目录 |
| 单文件数据库 | SQLite 一键备份 |
| 组件化前端 | Vue 3 SFC + Vuetify 3，140 个组件 |
| 国际化 | i18n 三语种（zh-CN / en-US / ru-RU）覆盖 |

## 六、经验与反思

### 6.1 做得好的

| 方面 | 表现 |
|------|------|
| 减法优先 | 砍掉 13 个平台适配 + 子 Agent 栈 + 多余主题，聚焦核心 |
| 先跑通再美化 | P0 先确保启动，P1 再上能力，P2 收敛品牌与文档 |
| 双 KB 解耦 | 早期决策用一套引擎承载两类语义，后期改动成本极低 |
| 文档同步迭代 | 每个阶段都有评审文档对应版本 |
| 构建产物入库 | 课设评审场景下 165 MB 换零部署成本，是合理权衡 |

### 6.2 可以改进

| 方面 | 改进方向 |
|------|----------|
| 早期单 commit 提交 | 应从一开始就语义化拆分，后期才补救 |
| 测试覆盖 | 集成测试为主，单元测试缺位 |
| 跨平台 | Linux/macOS 启动脚本待补 |
| Embedding 默认值 | 知识库依赖外部 embedding 是体验门槛 |
| 文档分发 | dashboard/dist 入库虽然方便评审，但仓库体积膨胀，长期需要权衡 |

### 6.3 反思

> 这个项目最让我意识到的一点是：**做加法很容易，做减法很难。**
>
> AstrBot 是一个非常优秀的多平台框架，但它的设计目标是"服务一切平台"。当我们把目标定为"AI 个人知识库助手"时，多平台、子 Agent 栈、多用户体系、插件市场等等，全部都是干扰项。每砍掉一个模块，剩下的代码反而更纯粹。
>
> 另一个体会是：**KB 和 Agent 不应该分两套系统。** 把对话沉淀的画像和外接资料用同一个检索引擎承载，让 Agent 工具直接调用同一个 KB Helper，整个系统的边界一下子就清楚了——一个能记住、能查、能动手的助手，本质上只需要这三件事的统一表达。
